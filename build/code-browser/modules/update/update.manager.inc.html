  13.  13  * archive file, or to directly upload the archive file. The archive is loaded
  14.  14  * into a temporary location, extracted, and verified. If everything is
  15.  15  * successful, the user is redirected to authorize.php to type in file transfer
  16.  16  * credentials and authorize the installation to proceed with elevated
  17.  17  * privileges, such that the extracted files can be copied out of the temporary
  18.  18  * location and into the live web root.

  20.  20  * Updating existing code is a more elaborate process. The first step is a
  21.  21  * selection form where the user is presented with a table of installed projects
  22.  22  * that are missing newer releases. The user selects which projects they wish to
  23.  23  * update, and presses the "Download updates" button to continue. This sets up a
  24.  24  * batch to fetch all the selected releases, and redirects to
  25.  25  * admin/update/download to display the batch progress bar as it runs. Each
  26.  26  * batch operation is responsible for downloading a single file, extracting the
  27.  27  * archive, and verifying the contents. If there are any errors, the user is
  28.  28  * redirected back to the first page with the error messages. If all downloads
  29.  29  * were extacted and verified, the user is instead redirected to
  30.  30  * admin/update/ready, a landing page which reminds them to backup their
  31.  31  * database and asks if they want to put the site offline during the update.

  33.  33  * authorize.php to supply their web root file access credentials. The
  34.  34  * authorized operation (which lives in update.authorize.inc) sets up a batch to
