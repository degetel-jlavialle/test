  14.  14  * Define all blocks provided by the module.
  15.  15  *
  16.  16  * This hook declares to Drupal what blocks are provided by your module and can
  17.  17  * optionally specify initial block configuration settings.
  18.  18  *
  19.  19  * In hook_block_info(), each block your module provides is given a unique
  20.  20  * identifier referred to as "delta" (the array key in the return value). Delta
  21.  21  * values only need to be unique within your module, and they are used in the
  22.  22  * following ways:
  23.  23  * - Passed into the other block hooks in your module as an argument to identify
  24.  24  *   the block being configured or viewed.
  25.  25  * - Used to construct the default HTML ID of "block-MODULE-DELTA" applied to
  26.  26  *   each block when it is rendered. This ID may then be used for CSS styling or
  27.  27  *   JavaScript programming.
  28.  28  * - Used to define a theming template suggestion of block__MODULE__DELTA, for
  29.  29  *   advanced theming possibilities.
  30.  30  * - Used by other modules to identify your block in hook_block_info_alter() and
  31.  31  *   other alter hooks.
  32.  32  * The values of delta can be strings or numbers, but because of the uses above
  33.  33  * it is preferable to use descriptive strings whenever possible, and only use a
  34.  34  * numeric identifier if you have to (for instance if your module allows users
  35.  35  * to create several similar blocks that you identify within your module code
  36.  36  * with numeric IDs). The maximum length for delta values is 32 bytes.
  37.  37  *
  38.  38  * @return
  39.  39  *   An associative array whose keys define the delta for each block and whose
  40.  40  *   values contain the block descriptions. Each block description is itself an
  41.  41  *   associative array, with the following key-value pairs:
  42.  42  *   - info: (required) The human-readable administrative name of the block.
  43.  43  *     This is used to identify the block on administration screens, and is not
  44.  44  *     displayed to non-administrative users.
  45.  45  *   - cache: (optional) A bitmask describing what kind of caching is
  46.  46  *     appropriate for the block. Drupal provides the following bitmask
  47.  47  *     constants for defining cache granularity:
  48.  48  *     - DRUPAL_CACHE_PER_ROLE (default): The block can change depending on the
  49.  49  *       roles the user viewing the page belongs to.
  50.  50  *     - DRUPAL_CACHE_PER_USER: The block can change depending on the user
  51.  51  *       viewing the page. This setting can be resource-consuming for sites with
  52.  52  *       large number of users, and should only be used when
  53.  53  *       DRUPAL_CACHE_PER_ROLE is not sufficient.
  54.  54  *     - DRUPAL_CACHE_PER_PAGE: The block can change depending on the page being
  55.  55  *       viewed.
  56.  56  *     - DRUPAL_CACHE_GLOBAL: The block is the same for every user on every page
  57.  57  *       where it is visible.
  58.  58  *     - DRUPAL_CACHE_CUSTOM: The module implements its own caching system.
  59.  59  *     - DRUPAL_NO_CACHE: The block should not get cached.
  60.  60  *   - properties: (optional) Array of additional metadata to add to the block.
  61.  61  *     Common properties include:
  62.  62  *     - administrative: Boolean that categorizes this block as usable in an
  63.  63  *       administrative context. This might include blocks that help an
  64.  64  *       administrator approve/deny comments, or view recently created user
  65.  65  *       accounts.
  66.  66  *   - weight: (optional) Initial value for the ordering weight of this block.
  67.  67  *     Most modules do not provide an initial value, and any value provided can
  68.  68  *     be modified by a user on the block configuration screen.
  69.  69  *   - status: (optional) Initial value for block enabled status. (1 = enabled,
  70.  70  *     0 = disabled). Most modules do not provide an initial value, and any
  71.  71  *     value provided can be modified by a user on the block configuration
  72.  72  *     screen.
  73.  73  *   - region: (optional) Initial value for theme region within which this
  74.  74  *     block is set. Most modules do not provide an initial value, and any value
  75.  75  *     provided can be modified by a user on the block configuration screen.
  76.  76  *     Note: If you set a region that isn't available in the currently enabled
  77.  77  *     theme, the block will be disabled.
  78.  78  *   - visibility: (optional) Initial value for the visibility flag, which tells
  79.  79  *     how to interpret the 'pages' value. Possible values are:
  80.  80  *     - BLOCK_VISIBILITY_NOTLISTED: Show on all pages except listed pages.
  81.  81  *       'pages' lists the paths where the block should not be shown.
  82.  82  *     - BLOCK_VISIBILITY_LISTED: Show only on listed pages. 'pages' lists the
  83.  83  *       paths where the block should be shown.
  84.  84  *     - BLOCK_VISIBILITY_PHP: Use custom PHP code to determine visibility.
  85.  85  *       'pages' gives the PHP code to use.
  86.  86  *     Most modules do not provide an initial value for 'visibility' or 'pages',
  87.  87  *     and any value provided can be modified by a user on the block
  88.  88  *     configuration screen.
  89.  89  *   - pages: (optional) See 'visibility' above. A string that contains one or
  90.  90  *     more page paths separated by "\n", "\r", or "\r\n" when 'visibility' is
  91.  91  *     set to BLOCK_VISIBILITY_NOTLISTED or BLOCK_VISIBILITY_LISTED (example:
